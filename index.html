一句话说清楚

传统蒙古文 Unicode 不是“一个视觉字形一个码位”。它保存的是名义字符序列，最终看到的独立形、词首形、词中形、词尾形、变体与竖排效果，由控制符、字体和排版引擎共同决定。

核心公式

字符编码 + 控制符 / 变体选择器 + OpenType 字体 + 排版引擎 = 正确显示

总览图：从输入到最终显示

这张流程图是理解传统蒙古文 Unicode 的主线：Unicode 负责“存什么”，字体和排版引擎负责“怎样显示”。

1

输入 / 键盘输入

输入法输出标准字符序列，不应该输出截图、私有码位，或只在某个软件里可用的假编码。

2

Unicode 码点序列

U+182E U+1823 U+1829 U+182D U+1823 U+182F

存储的是字符编号，不是最终字形外观。

3

控制符 / 辅助符号

FVS1 U+180BFVS2 U+180CFVS3 U+180DMVS U+180ENNBSP U+202FZWJ U+200DZWNJ U+200C
4

Shaping Engine / 排版引擎

根据前后文、连接状态、控制符和字体规则，决定独立形、词首形、词中形、词尾形及变体。

5

字体 / OpenType 字体

字体提供具体字形轮廓与替换规则，例如 cmap、GSUB、GPOS、竖排相关特性。

6

最终显示

ᠮᠣᠩᠭᠣᠯ

传统蒙古文通常竖排：从上到下书写，列从左到右排列。

Unicode 区块概览

  • U+1800–U+180A：标点符号
  • U+180B–U+180D：FVS1–FVS3
  • U+180E：MVS（历史兼容与规范演进相关）
  • U+1810–U+1819：数字
  • U+1820–U+1842：蒙古字母
  • U+1843–U+18AA：扩展字母与附加符号

为什么特别？

  • 文字竖排，不是普通横排脚本
  • 同一字母会随位置改变字形
  • 同一字符可有不同变体
  • 正确显示依赖字体 + 排版引擎

常见误区

  • Unicode ≠ 字体
  • 编码的是字符，不是图片或固定字形
  • 看到一样，不代表底层码点一样
  • 缺少字体时可能显示为方框或错形

完整解释：传统蒙古文 Unicode 到底解决什么？

1. 解决文本交换

Unicode 的第一价值是让传统蒙古文可以作为文本被保存、复制、搜索、传输和索引，而不是停留在图片、PDF 扫描件或某个软件私有编码里。

2. 不直接规定字形

Unicode 给字符分配码位，但不等于字体。最终字形外观由字体和渲染系统完成；同一码点在不同字体里可能显示风格不同。

3. 依赖上下文选形

传统蒙古文字母具有连写和位置变形，单个字符在词首、词中、词尾或独立状态下可能呈现不同外观。

重点：位置变化不是靠给每个视觉形态单独编码，而是靠“字符 + 上下文 + 排版规则 + 字体”生成。

1. 基础码位

每个字符都有 Unicode 编号。例如：ᠠ = U+1820，ᠨ = U+1828，ᠭ = U+182D。码位是文本交换的基础。

2. 变体与控制符

FVS 用于请求特定字形变体；ZWJ/ZWNJ 辅助连接或断开；NNBSP 常用于后缀前连接；MVS 属于历史兼容与规范演进中的关键点。

3. 渲染机制

文本先保存为码点序列，引擎读取上下文并选形，字体输出正确连写字形，最终在横排或竖排环境中显示。

4. 实践要点

不要把传统蒙古文当图片处理。编码、字体、排版、输入法、复制粘贴、搜索排序都要一起测试。

谁负责什么？

  • Unicode：给字符分配码位。
  • 输入法：输出标准字符序列。
  • 排版引擎：按上下文选形。
  • 字体：绘制最终字形。

正确显示链路

  • 文本保存为 Unicode 码点
  • 应用调用系统或浏览器排版能力
  • 排版引擎处理连接、变体和位置形
  • 字体提供可显示的 Mongolian glyph

失败时会怎样？

  • 显示方框 tofu
  • 字形不连写
  • 词首/词中/词尾形错误
  • 竖排方向、列方向或标点方向异常

工程判断：
1. 数据库保存 Unicode 码点序列，不保存截图。
2. 输入法输出标准字符，不输出私有编码。
3. 字体必须支持 Mongolian OpenType shaping。
4. 网页竖排优先测试 writing-mode: vertical-lr。
5. 搜索、复制、粘贴、排序、表单输入都要单独验收。
6. 同一视觉结果可能由不同码点序列产生，必须提供检测工具。
7. PDF、图片导出、OCR、LLM 输入都需要独立测试。
